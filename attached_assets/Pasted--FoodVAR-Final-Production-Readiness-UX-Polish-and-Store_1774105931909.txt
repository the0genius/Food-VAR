PHASE 1: ALLERGEN SAFETY, PRODUCT TRUST, AND RESULT-SCREEN CLARITY
==================================================

This is the highest priority area because FoodVAR is health-adjacent and user trust can be damaged quickly by confusing allergen behavior.

Current architecture already distinguishes:
- hard allergen alerts
- inferred allergen warnings
- product allergen states

Finish and polish this properly.

A. Complete allergen UX states on the result screen
Requirements:
- Red danger styling only for confirmed user-specific allergen alerts
- Amber or neutral styling for:
  - product contains allergens but not ones matching the user profile
  - possible risk inferred from ingredients
- Never use reassuring language that implies certainty
- Never use the word "safe"
- Do not allow contradictory messaging like:
  - reassuring message for the user
  - plus a severe red allergen-looking warning for a non-matching case

The result screen must clearly support these states:
1. hard_alert
   - Title like "Allergen Alert"
   - Message like "This product contains allergens that match your saved profile"
   - Red
2. product_contains_nonmatching
   - Title like "No match with your saved allergies"
   - Message like "This product still contains: X, Y"
   - Neutral or amber, not red
3. possible_risk
   - Title like "Possible allergen risk"
   - Message like "Ingredients suggest possible risk for: X, Y"
   - Amber
4. none
   - Either no allergen card
   - Or a subtle neutral informational row like "No allergen match found for your saved profile"

Apply this consistently to:
- fresh result screen
- history result rendering
- re-analysis result rendering

B. Define the FatSecret allergen certainty policy in code and UI
FatSecret products currently rely heavily on ingredient-derived and third-party data, which is useful but not equivalent to verified package-label allergen certainty.

Implement and document a clear policy:
- hard score-0 allergen alerts must continue to come only from declared or verified allergen data
- FatSecret ingredient-derived allergens can create warnings, not hard alerts
- when product source is third-party and allergen certainty is lower, the UI must communicate that carefully and clearly without clutter

Add a subtle trust note only when relevant, for example:
- "Verify package label for allergen certainty"
This must be shown only where needed and must feel premium, not scary or messy.

C. Begin deprecating the legacy mixed `allergens` field
The schema still carries legacy overlap between:
- allergens
- declaredAllergens
- inferredAllergens

Audit every current read/write path.
Goal:
- stop depending on generic `allergens` for display or safety logic
- prefer only:
  - declaredAllergens
  - inferredAllergens
- if `allergens` must remain for backward compatibility, fence it off clearly as legacy
- do not let it drive new UI behavior or new safety decisions

D. Tests
Add or update tests proving:
- hard alerts are user-specific and confirmed
- non-matching product allergens are shown without danger styling
- inferred warnings are softer than confirmed alerts
- legacy generic allergens do not create misleading UI or hard score-0 behavior

Acceptance criteria:
- allergen communication feels precise, trustworthy, and non-contradictory
- users can distinguish between personal danger, product contents, and possible risk
- legacy allergen ambiguity is being phased out
